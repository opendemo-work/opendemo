# SRE Fundamentals

站点可靠性工程(SRE)核心原则与实践入门。

## 什么是SRE

SRE是Google提出的运维理念，通过软件工程方法解决运维问题：

```
DevOps vs SRE:
┌─────────────────────────────────────────────────────────┐
│  DevOps                    SRE                          │
│  ───────                   ───                          │
│  文化运动                  具体实践                      │
│  打破团队墙                软件工程化运维                 │
│  持续交付                  可靠性量化                    │
│  自动化                    错误预算驱动                  │
└─────────────────────────────────────────────────────────┘
```

## SRE核心原则

### 1. 拥抱风险
```
可靠性目标:
服务等级    每年可接受停机时间
────────    ─────────────────
99%         3.65天
99.9%       8.76小时
99.99%      52.56分钟
99.999%     5.26分钟

关键认知: 100%可靠性是不切实际且昂贵的
```

### 2. 服务水平目标(SLO)
```yaml
# SLO定义示例
service: payment-api
slos:
  - name: availability
    target: 99.95%
    window: 30d
    burn_rate_alert: 2x
    
  - name: latency-p99
    target: 200ms
    window: 30d
    
  - name: error-rate
    target: 0.1%
    window: 30d
```

### 3. 错误预算
```
错误预算计算:
SLO目标: 99.9% (30天)
允许错误: 0.1% × 30天 = 43.2分钟/月

错误预算消耗:
├── 正常发布: 消耗5-10%
├── 大型变更: 消耗20-30%
├── 事故恢复: 消耗50%+
└── 预算耗尽: 暂停发布，专注可靠性改进
```

## 关键指标

| 指标 | 描述 | 计算方式 |
|------|------|----------|
| MTTR | 平均恢复时间 | 故障发现到恢复的平均时间 |
| MTBF | 平均故障间隔 | 两次故障之间的平均时间 |
| MTTD | 平均检测时间 | 故障发生到被发现的平均时间 |
| Error Budget | 错误预算 | 100% - SLO目标 |
| Burn Rate | 消耗速率 | 当前错误率/预算错误率 |
